Transport¶

template<typename T>
concept TransportTrait¶
#include “core/Transport.hpp”

Transport concept. The representation of the actor model used by this project.

Generally, TransportTrait can be considered as an extended version of original spexpaxos’s Transport interface. Because it is a concept rather than a superclass, the name Transport is intended to be preserved as protocol classes’ template parameter.

Note

Despite the name, TransportReceiver is not a necessary part of the abstraction, and transport does not depend on it.

template<typename Self>
struct oskr::TransportBase¶

Common complement of TransportTrait concept. Including re-export and derivation of types and serveral default implementations. All methods are not virtual be design, because the whole Transport is a compile-time polymorphism.

using Receiver = Fn<void(const Address &remote, Desc descriptor)>¶

Receiver closure. Related states should be captured by needed.

The closure will be called on every packet-receiving. The remote reference is alive as long as transport object is alive. The provided descriptor keeps the underlying message in scope. Although receiver is allowed to keep descriptor after returning from this callback, receiver should release descriptor as soon as possible, make sure to maintain a upper bound of number of kept description to prevent transport’s descriptor get exhausted.

Warning

Receiver must put as lightest as possible processing logic in this callback, do only the most basic works like spawn or even nothing (e.g., drop the packet), and return quickly.

Warning

This is required because receiver is called from RX thread, which is unacceptable to be blocked by protocol logic. For certain transport (e.g., DPDK-based), if RX interval is too large, incoming packets will be dropped in network interface, and such situation will be considered as fatal error. Understand what you are doing in receiver.

using Callback = FnOnce<void()>¶
using Write = Fn<std::size_t(TxSpan<buffer_size> buffer)>¶

The common argument of sendMessage* methods, write message content into buffer, return written length, which should not be greater than buffer_size.

Transport promises that this callback will not be accessed any more after returning from sendMessage*. It is safe to capture local objects by reference.

 14/*! @brief Common complement of `TransportTrait` concept. Including re-export
 15and derivation of types and serveral default implementations. All methods are
 16not virtual be design, because the whole `Transport` is a compile-time
 17polymorphism.
 18*/
 19template <typename Self> struct TransportBase {
 20    using Address = typename TransportMeta<Self>::Address;
 21    using Desc = typename TransportMeta<Self>::Desc;
 22    static constexpr std::size_t buffer_size = TransportMeta<Self>::buffer_size;
 23
 24    //! Receiver closure. Related states should be captured by needed.
 25    //!
 26    //! The closure will be called on every packet-receiving. The `remote`
 27    //! reference is alive as long as transport object is alive. The provided
 28    //! `descriptor` keeps the underlying message in scope. Although receiver is
 29    //! allowed to keep `descriptor` after returning from this callback,
 30    //! receiver should release `descriptor` as soon as possible, make sure to
 31    //! maintain a upper bound of number of kept `description` to prevent
 32    //! transport's descriptor get exhausted.
 33    //!
 34    //! @warning Receiver must put as lightest as possible processing logic in
 35    //! this callback, do only the most basic works like `spawn` or even nothing
 36    //! (e.g., drop the packet), and return quickly.
 37    //! @warning This is required because receiver is called from RX thread,
 38    //! which is unacceptable to be blocked by protocol logic. For certain
 39    //! transport (e.g., DPDK-based), if RX interval is too large, incoming
 40    //! packets will be dropped in network interface, and such situation will be
 41    //! considered as **fatal error**. Understand what you are doing in
 42    //! receiver.
 43    using Receiver = Fn<void(const Address &remote, Desc descriptor)>;
 44    using Callback = FnOnce<void()>;
 45
 46    //! The common argument of `sendMessage*` methods, write message content
 47    //! into `buffer`, return written length, which should not be greater than
 48    //! `buffer_size`.
 49    //!
 50    //! Transport promises that this callback will not be accessed any more
 51    //! after returning from `sendMessage*`. It is safe to capture local objects
 52    //! by reference.
 53    using Write = Fn<std::size_t(TxSpan<buffer_size> buffer)>;
 54
 55    template <typename Sender>
 56    void sendMessageToReplica(
 57        const Sender &sender, ReplicaId replica_id, Write write)
 58    {
 59        Self &transport = static_cast<Self &>(*this);
 60        transport.sendMessage(
 61            sender, transport.config.replica_address_list[replica_id], write);
 62    }
 63
 64    //! Practical transport implementation should override this with a more
 65    //! efficient version that only call `write` once.
 66    template <typename Sender>
 67    void sendMessageToAll(const Sender &sender, Write write)
 68    {
 69        Self &transport = static_cast<Self &>(*this);
 70        for (auto dest : transport.config.replica_address_list) {
 71            if (dest != sender.address) {
 72                transport.sendMessage(sender, dest, write);
 73            }
 74        }
 75    }
 76
 77    template <typename Sender>
 78    void sendMessageToMulticast(const Sender &sender, Write write)
 79    {
 80        Self &transport = static_cast<Self &>(*this);
 81        transport.sendMessage(
 82            sender, transport.config.multicast_address, write);
 83    }
 84};

 90/*! @brief Transport concept. The representation of the actor model used by this
 91project.
 92
 93* @note Despite the name, `TransportReceiver` is not a necessary part of the
 94abstraction, and transport does not depend on it.
 95
 96Generally, `TransportTrait` can be considered as an extended version of original
 97spexpaxos's `Transport` interface. Because it is a concept rather than a
 98superclass, the name `Transport` is intended to be preserved as protocol
 99classes' template parameter.
100*/
101template <typename T>
102concept TransportTrait =
103    //! TransportMeta specialization.
104    //!
105    requires
106{
107    //! The address type used by transport implementation. The type should has
108    //! value semantic, and avoid heap allocation.
109    typename TransportMeta<T>::Address;
110    typename TransportMeta<T>::Desc;
111    typename std::integral_constant<std::size_t, TransportMeta<T>::buffer_size>;
112}
113//! Subclassing `TransportBase`
114//!
115&&std::derived_from<T, TransportBase<T>>
116    //! Receiving interfaces
117    //!
118    &&requires(
119        T transport, const typename T::Address address,
120        typename T::Receiver receiver)
121{
122    //! Register a receiver closure. The closure will be called whenever message
123    //! arriving at `address`.
124    transport.registerReceiver(address, receiver);
125    // TODO multicast
126}
127//! Scheduling interfaces
128//!
129&&requires(T transport, typename T::Callback callback)
130{
131    //! Spawn a sequential task. All sequential tasks will be executed without
132    //! temporal overlapping, so it is safe to update states without locking.
133    transport.spawn(std::move(callback));
134}
135//! Sending interfaces
136//!
137&&requires(
138    T transport, const SenderArchetype<T> &sender,
139    const typename T::Address &dest, typename T::Write write)
140{
141    //! Allocate a TX message buffer with `buffer_size` space, call `write` with
142    //! it to serialize message into the buffer, then send message to `dest`.
143    //!
144    //! `sender` should probably be `*this`, but anything with a public
145    //! `address` should be good.
146    transport.sendMessage(sender, dest, write);
147};
