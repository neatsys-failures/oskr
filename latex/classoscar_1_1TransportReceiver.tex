\doxysubsection{Detailed Description}
\subsubsection*{template$<$typename Transport$>$\newline
class oscar\+::\+Transport\+Receiver$<$ Transport $>$}

The base class for packet-\/listeners, i.\+e., all participants of protocols. The receiving half of the actor model. 

Each instance of {\ttfamily \mbox{\hyperlink{classoscar_1_1TransportReceiver}{Transport\+Receiver}}} has a static address throughout its lifetime. It also can only work with one specific {\ttfamily \mbox{\hyperlink{classoscar_1_1Transport}{Transport}}} type.

After registered to a {\ttfamily \mbox{\hyperlink{classoscar_1_1Transport}{Transport}}} instance, virtual method {\ttfamily receive\+Message} will be called whenever messages arrives receiver\textquotesingle{}s address.

The {\ttfamily \mbox{\hyperlink{classoscar_1_1TransportReceiver}{Transport\+Receiver}}} itself does not make any assumption on how to process received message, but a subclass will probably want to call {\ttfamily \mbox{\hyperlink{classoscar_1_1Transport}{Transport}}} methods to send message and schedule timeout. So it is conventional to ask for a reference of {\ttfamily \mbox{\hyperlink{classoscar_1_1Transport}{Transport}}} instance when constructing subclass. Because the {\ttfamily \mbox{\hyperlink{classoscar_1_1Transport}{Transport}}} base class follows CRTP, {\ttfamily \mbox{\hyperlink{classoscar_1_1Transport}{Transport}} \&} is basically equals to {\ttfamily \mbox{\hyperlink{classoscar_1_1Transport}{oscar\+::\+Transport}}$<$\mbox{\hyperlink{classoscar_1_1Transport}{Transport}}$>$ \&} in every situation.
